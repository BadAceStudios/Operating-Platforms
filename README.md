# Software Design Portfolio

## Project Summary

The Gaming Room was the client for this project.  They had an existing Android game called *Draw It or Lose It* and wanted to expand it into a web-based application that could support multiple operating platforms.  The software needed to support multiple games, teams, and players while maintaining unique names and centralized management of game data.

## What I Did Well

I think I did particularly well with organizing the software design document and connecting the client’s requirements to real design decisions.  I was able to evaluate different platforms, compare their strengths and weaknesses, and recommend a Linux-based server platform based on scalability, cost, security, and support for distributed systems.

## Helpful Parts of the Design Process

Working through the design document helped me better understand the system before focusing on code.  Breaking the project into requirements, constraints, domain models, platform evaluations, and recommendations made it easier to see how each design decision affected the overall application.  It also helped me think more like a developer working for a client instead of only thinking about the code itself.

## What I Would Revise

If I could revise one part of my work, I would continue improving the recommendations section with even more specific implementation details.  For example, I could include more details about cloud deployment, database scaling, and monitoring tools.  This would make the document even more useful for a development team preparing to build the application.

## Interpreting User Needs

I interpreted the client’s needs by focusing on what The Gaming Room wanted the application to accomplish: multi-platform access, scalability, reliable performance, and secure user data.  Considering user needs is important because software should solve the client’s actual problem, not just use technology for its own sake.  If user needs are not considered, the final system may work technically but fail to support the people and business it was designed for.

## Software Design Approach

My approach to designing the software was to first understand the client’s requirements, then evaluate the best architecture and platform options to support those needs.  I used object-oriented design principles, design patterns, platform comparison, and system architecture analysis to guide the design.  In the future, I would use the same strategy by starting with requirements, identifying constraints, comparing possible solutions, and then choosing the design that best supports the client’s goals.
